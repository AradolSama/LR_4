6. Каким образом изменяются версии расширенных систем и контролю версій?
Хороша розподілена модель, оскільки кожен розробник отримує локальне сховище з повною история комітів,
що робить git быстрым порождением с другими VC. Можливості розгалуження и злиття прості (оскільки дешеві),
хороша цілісність даних. Вони бесплатно и с відкритим кодом, ми можно легко загрузить виходний код
і внести до нього зміни. Вони можуть ефективно виконувати великі проекти. Операции "толкать / тянуть" быстрее для простого.
Вони економлять час, а розробники можуть отримувати та створювати запити на витяг без перемикання .. Надлишковість
даних и реплікації. Дополнения можуть бути написані багатьма мовами

7. Які недоліки по установке розполнения системи контролю версій? 
Головним недоліком протоколу Git є відсутність авторизації .GIT вимагає технічної досконалості,
а на Windows він працює повільніше. Вони мають виснажливі командні рядки для введения
і не відстежують перейменування. Вони мають поганий графічний интерфейс и удобство воспроизведения.
Крім того, вони забирають багато ресурсів, що уповільнює продуктивність. GIT не підтримує перевірку
піддерев. Для кожного проекта центральная служба потрібно налаштуват для кількох сховищ пакетів.Немає
підтримки вікон и не відстежує порожні папки. Немає вбудованого контролю доступу та не підтримує
 двійкові файли.Вони не передбачають механізмів контролю доступу в разі безпеки.

9. Навіщо потрібні розподілені системи?
Git содержит файлы, которые могут быть внесены в файли, и могут записывать файлы, которые были созданы,
і в можете повернутися до певних версій, якщо вам знадобиться. Git також спрощує співпрацю,
дозволяючи всех людей, внесенных в список людей, об'єднувати в одне джерело
